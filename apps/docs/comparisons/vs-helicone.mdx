## Migration

```bash
tlane migrate helicone --apply
```

This scans your `.env*` files and SDK call sites, then rewrites the Helicone base URL and
auth headers (`Helicone-Auth`) to their Tracelane equivalents — a config + environment
migration, shown as a diff before writing. It does **not** import historical trace data.
Most teams cut over the same afternoon; new traces flow from the moment you redeploy.

## FAQ

<AccordionGroup>

<Accordion title="Does Tracelane copy any Helicone code?">
No. Helicone's `ai-gateway` is GPL-3.0. Copying any of it would taint the Apache 2.0 Tracelane stack with viral copyleft. We reference patterns at most; we copy zero code. CLAUDE.md banned-dependency list enforces this.
</Accordion>

<Accordion title="Is Tracelane feature-equivalent to Helicone?">
Tracelane covers 30+ providers, BYOK with envelope encryption, usage/token tracking, OTel GenAI semconv, and a full Apache 2.0 self-host. It adds inline predictive guardrails, tamper-evident audit ledger, and the Prompt Promotion workflow Helicone does not ship.
</Accordion>

<Accordion title="What happens to my historical Helicone data?">
Your historical traces stay where they are — `tlane migrate helicone` is a config + environment migration, not a trace-data importer, so it does not pull your past Helicone traces across. New traces flow into Tracelane from the moment you redeploy. A historical-trace importer (Helicone JSON dump → OTel GenAI semconv) is on the roadmap; until it ships, export from Helicone for archival and start fresh in Tracelane.
</Accordion>
