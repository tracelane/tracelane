# Audit conformance statement

**Effective date:** 2026-06-16 (V1 launch). **Coverage:** every
workspace with the **Audit add-on SKU** (`f_audit_addon = TRUE` in
`workspace_entitlements`).

Tracelane attests that the Audit add-on SKU satisfies the EU AI Act
Article 12 recordkeeping obligations for high-risk AI systems
deployed on the Tracelane platform. This statement is the written
artefact intended for:

- Internal compliance teams preparing an Article 12 attestation.
- External auditors performing pre-engagement diligence.
- Market-surveillance authorities exercising Article 12(3)(c)
  inspection rights.

For the underlying cryptographic mechanisms, see
[ADR-018](https://github.com/tracelane/tracelane/blob/main/decisions/ADR-018-tamper-evident-ledger-v1.md)
and [ADR-034](https://github.com/tracelane/tracelane/blob/main/decisions/ADR-034-audit-verifier-cli.md).

> **Status — what is live vs. on the roadmap.** The **per-tenant
> SHA-256 hash chain is live and universal** (every workspace,
> including the free tier) and is independently verifiable today by
> recomputing the chain offline. **Ed25519 Merkle-root batch signing,
> Merkle aggregation, and Sigstore Rekor v2 public-transparency
> anchoring are now live in production** for Audit-add-on workspaces:
> each batch's Merkle root is Ed25519-signed and best-effort anchored
> to the public Sigstore Rekor v2 log
> ([`log2025-1.rekor.sigstore.dev`](https://log2025-1.rekor.sigstore.dev)),
> and each anchor's inclusion proof + signed checkpoint are captured so
> a regulator can re-verify **offline** against the public log. **eIDAS
> QTSP timestamping remains on the roadmap.** Two honest scope notes:
> (1) anchoring is **best-effort** — if the public log is unreachable a
> batch stays signed-locally-but-not-yet-anchored, and the hash chain +
> Ed25519 signature still hold (the tamper-evidence guarantee never
> depends on the external log being up); (2) the chain today covers
> **gateway-proxied** events (LLM provider calls + guardrail verdicts
> routed through the Tracelane gateway) — spans captured via the
> SDK/OTLP self-host path are recorded full-fidelity but are **not yet
> chained or anchored** (batch-digest coverage of that path is the next
> increment). The tamper-evidence guarantee rests on the live hash
> chain + offline recompute; the Ed25519 signature and Rekor inclusion
> proof are additional, independently-verifiable layers on top.

For the regulator-runnable verifier, see [`/audit/verifier-cli`](/audit/verifier-cli).

## Statement

For each high-risk AI system deployed against a Tracelane workspace
with the Audit add-on SKU active, Tracelane represents and warrants
that:

### Article 12(2)(a) — automatic log generation

Every runtime event the deployed AI system emits to the Tracelane
gateway — including LLM provider calls, tool invocations,
guardrail-layer decisions, agent-to-agent handoffs, and MCP
invocations — is written into a per-tenant tamper-evident audit
ledger. Writes are synchronous with the agent event; no asynchronous
deferral that could be selectively suppressed exists. The
implementation is open-source under Apache-2.0 at
`crates/gateway/src/audit.rs` and is independently verifiable.

### Article 12(2)(b) — risk-situation identification

Every audit row carries a structured `event_type` enum, the
identifying `actor` of the runtime component, the full canonical
`payload` (subject to the PII-redaction policy at
`crates/policy/src/pii.rs`), and the OTel-GenAI semantic-convention
attributes for LLM provider calls (model, token counts, intervention
decisions). A regulator-side query "every time the system blocked a
user request and the reason" returns in sub-second time against any
Tracelane workspace.

### Article 12(2)(c) — six-month retention minimum

Tracelane hardcodes `AUDIT_ADDON_MIN_RETENTION_DAYS = 180` in
`crates/gateway/src/audit_retention.rs`. The `resolve_audit_retention`
function panics fail-loud if any caller attempts to set a
contractual override shorter than 180 days for an Audit-add-on
workspace. This is enforced *in source*, not only in configuration
— a configuration error cannot accidentally shorten retention below
the floor.

Workspaces without the Audit add-on are subject to the base-tier
retention from
[ADR-020](https://github.com/tracelane/tracelane/blob/main/decisions/ADR-020-pricing-v2-honest-caps.md)
(free 7d / builder 30d / team 90d / business 180d / enterprise 365d)
and are **not** covered by this conformance statement.

### Article 12(3)(a) — log conformity with established procedures

The audit-ledger format (row hash function, per-tenant chain
construction, Merkle aggregation, Rekor anchor procedure) is
documented in [ADR-018](https://github.com/tracelane/tracelane/blob/main/decisions/ADR-018-tamper-evident-ledger-v1.md)
(license: CC0 — public domain). Three independent reference
verifiers (Rust, Python, TypeScript) at `packages/verifier-{rust,
python,typescript}/` produce byte-identical `VerifyReport` JSON for
the same input, demonstrating that the procedure is
specification-driven, not implementation-locked. The live conformity
check is the per-tenant SHA-256 hash-chain replay, the per-batch
Ed25519 Merkle-root signature, and — for anchored batches — the
Sigstore Rekor v2 inclusion proof, all verified offline.

Tracelane operates **no private fork** of Sigstore Rekor: each
Merkle-root anchor is published to the public
[`log2025-1.rekor.sigstore.dev`](https://log2025-1.rekor.sigstore.dev)
transparency log operated by the Linux Foundation Open Source Security
Foundation (OpenSSF), so a regulator can re-verify each anchor
independently — offline against the bundled inclusion proof + signed
checkpoint — without trusting any Tracelane HTTPS endpoint. Public
anchoring **is operating in production** (first anchor: log index
`19398597`, 2026-07-13).

**Forgery resistance (why a real Rekor entry is not enough).** Rekor v2
is a *permissionless* log: anyone can submit any well-formed entry and
receive a real inclusion proof + checkpoint. A "real Rekor entry" alone
therefore proves only that *some* body was admitted to the log — not
that **Tracelane** admitted it. Tracelane's verifier closes this gap: it
accepts an anchor only when the batch's Ed25519 signature verifies
against the workspace's **own** public key (delivered out-of-band, not
taken from the anchor bundle), and that signature cryptographically
binds the anchor's ECDSA key, anchor state, and log index. The published
conformance vectors include `forged-anchor.ndjson` — a genuine,
publicly-queryable Rekor entry planted by an attacker's own key over a
tampered chain — and all three reference verifiers **reject** it while
accepting the legitimately-anchored `anchored.v1.ndjson`.

### Article 12(3)(b) — operator log access

`GET /api/audit/range?workspace=<uuid>&from=<iso>&to=<iso>` returns
the NDJSON ledger range for the queried workspace. Access requires
an `audit:read`-scoped API key issued by Tracelane support and
bound to the calling workspace. Keys are issued with a documented
time bound (default 30 days; longer on contract).

### Article 12(3)(c) — market-surveillance authority access

Under formal authority, market-surveillance representatives request
a time-bounded `audit:read` key scoped to the workspace under
inspection through `security@tracelane.dev`. Tracelane issues the
key within one business day. Authorities run `tracelane-audit
verify` from the published static binary; the verifier is
Cosign-signed via GitHub Actions OIDC and bears SLSA Build Level 3
provenance, so the binary itself requires no trust in any
Tracelane-operated infrastructure to verify. The checks the authority
runs are all offline (no Tracelane endpoint needed): the hash-chain
replay, the per-batch Ed25519 signature against the workspace's own
public key, and — for anchored batches — the Sigstore Rekor v2
inclusion proof + signed checkpoint against the pinned public-log key.
These Rekor proofs are produced in production today; the authority
re-verifies each anchor against the public transparency log entirely
offline from the exported bundle.

### Article 12(3)(d) — automatic prevention of unauthorised access

Tenant isolation is enforced at three independent layers, documented
in [ADR-031](https://github.com/tracelane/tracelane/blob/main/decisions/ADR-031-admin-audit-and-ch-resource-caps.md)
and [`/security/tenant-isolation`](/security/tenant-isolation):

1. ClickHouse-side: every query carries a `WHERE tenant_id = ?`
   filter (CI grep blocks any new query that omits it) plus per-tier
   resource caps.
2. R2-side: every object key is prefixed `tenants/<workspace_uuid>/`
   with a debug-`assert!` + release-refuse on prefix violation.
3. Postgres-side: row-level + admin-audit-log surfaces.

## Limitations and V1.1 follow-ups

This conformance statement covers V1 launch. The following items
ship in V1.1 (target: M+1 to M+3 post-launch). None are required to
satisfy Article 12 obligations at V1 launch; all strengthen the
posture or close gaps already-acceptable-under-the-Act:

- **Automated audit-data retention cleanup job.** V1 does not delete
  audit data; the 180-day floor is trivially satisfied. V1.1 ships
  the cron + DELETE pipeline using `resolve_audit_retention`.
- **PDF conformance packet export.** `tracelane-audit export
  --format pdf` ships in V1.1. V1 produces the equivalent
  human-readable content via `--format text` and `--format json`.
- **AUDIT-001 per-tenant chain isolation hardening.** V1's per-tenant
  `seq` + per-tenant Ed25519 keypair already enforce per-tenant
  cryptographic integrity. V1.5 ships the per-tenant chain
  `DashMap<TenantId, ChainState>` refactor for additional defence in
  depth against operator-side errors.
